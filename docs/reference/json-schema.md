# JSON Output Schema

This document describes the structure of JSON output files generated by ipc-benchmark.

## Summary JSON (`-o` flag)

### Top-Level Structure

```json
{
  "metadata": { ... },
  "results": [ ... ],
  "summary": { ... }
}
```

### Metadata Object

```json
{
  "metadata": {
    "version": "0.1.0",
    "timestamp": "2024-01-01T12:00:00.000Z",
    "total_tests": 4,
    "system_info": {
      "os": "linux",
      "architecture": "x86_64",
      "cpu_cores": 8,
      "memory_gb": 16.0,
      "rust_version": "1.75.0",
      "benchmark_version": "0.1.0"
    }
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `version` | string | Benchmark suite version |
| `timestamp` | string | ISO 8601 timestamp |
| `total_tests` | integer | Number of mechanism tests run |
| `system_info.os` | string | Operating system |
| `system_info.architecture` | string | CPU architecture |
| `system_info.cpu_cores` | integer | Number of CPU cores |
| `system_info.memory_gb` | float | System memory in GB (best-effort; may be a placeholder) |

### Results Array

Each element represents one mechanism test:

```json
{
  "results": [
    {
      "mechanism": "UnixDomainSocket",
      "test_config": { ... },
      "one_way_results": { ... },
      "round_trip_results": { ... },
      "summary": { ... }
    }
  ]
}
```

### Test Configuration Object

```json
{
  "test_config": {
    "message_size": 1024,
    "buffer_size": 8192,
    "concurrency": 1,
    "msg_count": 10000,
    "duration": null,
    "one_way_enabled": true,
    "round_trip_enabled": true,
    "warmup_iterations": 1000,
    "percentiles": [50.0, 95.0, 99.0, 99.9]
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `message_size` | integer | Message payload size in bytes |
| `buffer_size` | integer | Buffer size for mechanism |
| `concurrency` | integer | Number of concurrent workers |
| `msg_count` | integer | Messages sent (if count-based) |
| `duration` | string/null | Test duration (if duration-based) |
| `warmup_iterations` | integer | Warmup messages |
| `percentiles` | array[float] | Requested percentiles |

### Latency Results Object

Both `one_way_results` and `round_trip_results` share this structure:

```json
{
  "one_way_results": {
    "latency": {
      "latency_type": "OneWay",
      "min_ns": 1500,
      "max_ns": 45000,
      "mean_ns": 3200.5,
      "median_ns": 3100,
      "std_dev_ns": 1250.3,
      "percentiles": [
        {"percentile": 50.0, "value_ns": 3100},
        {"percentile": 95.0, "value_ns": 5200},
        {"percentile": 99.0, "value_ns": 8500},
        {"percentile": 99.9, "value_ns": 12000}
      ]
    },
    "throughput": {
      "messages_per_second": 312500.0,
      "bytes_per_second": 320000000.0,
      "total_messages": 10000,
      "total_bytes": 10240000,
      "duration_ns": 32000000
    }
  }
}
```

### Latency Object

| Field | Type | Description |
|-------|------|-------------|
| `latency_type` | string | "OneWay" or "RoundTrip" |
| `min_ns` | integer | Minimum latency in nanoseconds |
| `max_ns` | integer | Maximum latency in nanoseconds |
| `mean_ns` | float | Mean latency in nanoseconds |
| `median_ns` | integer | Median latency (P50) |
| `std_dev_ns` | float | Standard deviation |
| `percentiles` | array | Percentile values |

### Percentile Object

```json
{
  "percentile": 99.0,
  "value_ns": 8500
}
```

### Throughput Object

| Field | Type | Description |
|-------|------|-------------|
| `messages_per_second` | float | Message rate |
| `bytes_per_second` | float | Data rate |
| `total_messages` | integer | Total messages sent |
| `total_bytes` | integer | Total bytes transferred |
| `duration_ns` | integer | Test duration in nanoseconds |

### Result Summary Object

```json
{
  "summary": {
    "total_messages_sent": 20000,
    "total_bytes_transferred": 20480000,
    "average_throughput_mb_s": 305.17,
    "p95_latency_ns": 5200,
    "p99_latency_ns": 8500
  }
}
```

### Top-Level Summary

```json
{
  "summary": {
    "fastest_mechanism": "SharedMemory",
    "lowest_latency_mechanism": "SharedMemory"
  }
}
```

## Streaming JSON (`--streaming-output-json`)

Columnar format for per-message data:

```json
{
  "headings": [
    "timestamp_ns",
    "message_id",
    "mechanism",
    "message_size",
    "one_way_latency_ns",
    "round_trip_latency_ns"
  ],
  "data": [
    [1737417600000000000, 1, "UnixDomainSocket", 1024, 50000, null],
    [1737417600000100000, 2, "UnixDomainSocket", 1024, 51000, null],
    [1737417600000200000, 3, "UnixDomainSocket", 1024, null, 98000]
  ]
}
```

### Headings

| Column | Type | Description |
|--------|------|-------------|
| `timestamp_ns` | integer | Unix epoch timestamp in nanoseconds for when the record was captured |
| `message_id` | integer | Sequential message number |
| `mechanism` | string | Mechanism name (e.g. "UnixDomainSocket", "TcpSocket") |
| `message_size` | integer | Message payload size in bytes |
| `one_way_latency_ns` | integer/null | One-way latency in nanoseconds (null if round-trip-only record) |
| `round_trip_latency_ns` | integer/null | Round-trip latency in nanoseconds (null if one-way-only record) |

### Timestamp and Clock Semantics

- **`timestamp_ns` is epoch time** (nanoseconds since Unix epoch). It is intended
  for correlating measurements with external logs/telemetry.
- **Latency values are durations** derived from **monotonic** timing inside the
  benchmark and are not affected by NTP/manual clock changes.

## Streaming CSV (`--streaming-output-csv`)

The streaming CSV output uses the **same columns and ordering** as the streaming
JSON `headings` array:

```csv
timestamp_ns,message_id,mechanism,message_size,one_way_latency_ns,round_trip_latency_ns
1737417600000000000,1,UnixDomainSocket,1024,50000,
1737417600000100000,2,UnixDomainSocket,1024,,98000
```

Notes:
- Fields that are not applicable for a given record are **blank** in CSV (and
  `null` in JSON).

## Mechanism Values

| Value | Description |
|-------|-------------|
| `UnixDomainSocket` | Unix Domain Socket |
| `SharedMemory` | Shared Memory (ring buffer) |
| `TcpSocket` | TCP Socket |
| `PosixMessageQueue` | POSIX Message Queue |

Notes:
- The `mechanism` field is derived from the CLI mechanism selection (`-m`), so
  `--shm-direct` still reports `SharedMemory` as the mechanism name. The direct
  memory mode is captured by configuration/flags, not by a distinct mechanism
  enum value.

## System Information Notes

- `system_info.memory_gb` is currently a best-effort/placeholder value. It is
  not guaranteed to reflect the true installed memory on all platforms.
- `system_info.rust_version` reflects the packageâ€™s declared Rust version
  (MSRV) from build metadata, not necessarily the exact `rustc --version` used
  to compile the binary.

## Example: Processing with jq

```bash
# Get mean one-way latency for first mechanism
cat results.json | jq '.results[0].one_way_results.latency.mean_ns'

# Get all P99 latencies
cat results.json | jq '.results[].one_way_results.latency.percentiles[] | select(.percentile == 99.0) | .value_ns'

# Get throughput in MB/s
cat results.json | jq '.results[0].summary.average_throughput_mb_s'
```

## Example: Processing with Python

```python
import json

with open('results.json') as f:
    data = json.load(f)

for result in data['results']:
    mechanism = result['mechanism']
    mean_ns = result['one_way_results']['latency']['mean_ns']
    print(f"{mechanism}: {mean_ns/1000:.2f} Âµs")
```
